## Options considered

1. **Per-issue commits** — one commit per ISSUE-NNNN. Maximum bisect granularity; ~14 commits for
   Medium + elevated-Low scope. High commit noise; same-area changes lose context.

2. **Cluster-by-area commits** — one commit per logical cluster (A: ui-primitives, B: dashboard-errors,
   C: backend, D: sdk, E: a11y, F: loading-consistency). 6 commits. Natural dependency order.

3. **Single commit** — entire polish stage. Unacceptable per CLAUDE.md atomic commit rule.

## Decision

Use **Option 2 — cluster-by-area commits** (Q-1.1-POLISH-2 resolution).

## Rationale

Cluster batching matches the dependency order (Cluster A produces primitives that Cluster B
consumes), keeps each commit independently reviewable, and avoids micro-commit noise for
changes that share context (all `role="alert"` fixes in one a11y commit; all `staleTime`
additions in one sdk commit). Consistent with brief §4 commit discipline.

## Decisions (Q-1.1-POLISH-* resolutions)

1. **Q-1.1-POLISH-1 (token scope):** minimal — fix class inconsistencies using existing tokens;
   no new `tokens.css` entries.

2. **Q-1.1-POLISH-2 (commit batching):** cluster by area — 6 clusters (A–F), one commit per
   cluster.

3. **Q-1.1-POLISH-3 (mobile scope):** student-facing only — teacher surfaces are desktop-primary.

4. **Q-1.1-POLISH-4 (axe-core gating):** carry-to-preview — ISSUE-0038 gates the preview deploy
   stage; no code action in polish stage.

5. **Q-1.1-POLISH-5 (Low elevations):** elevate ISSUE-0046 + ISSUE-0065 → Cluster E (a11y);
   ISSUE-0064 + ISSUE-0047 → Cluster F (loading-consistency). Defer ISSUE-0044 + ISSUE-0066.

6. **Q-1.1-POLISH-6 (ISSUE-0067 gate):** carry — add `NODE_OPTIONS=--use-system-ca` note to
   README; stage does not block on local build. CI/Vercel unaffected.

7. **Q-1.1-POLISH-7 (ISSUE-0039 + 0045 placement):** confirmed in-scope as Medium. ISSUE-0039
   folds into Cluster B (depends on Cluster A UpgradeState). ISSUE-0045 folds into Cluster E.

8. **Cluster execution order:** A → (B + D parallel) → C → E → F → matrix sweep.

9. **T5 gate model per cluster:**
   - Cluster A (ui-primitives, new `ErrorState` + `UpgradeState`): full T5 three-gate
     (sketch → operator approval → skeleton checkpoint → fill).
   - Cluster B (dashboard-errors): per-surface. Student dashboard 6-widget isError pass: T5
     three-gate. Forms/session-selection 402 wiring (ISSUE-0039): direct fix per brief §4
     (single-file, clear from issue text).
   - Clusters C, D, E: direct fix-and-verify (brief §4 trivial isolated fix — no layout change).
   - Cluster F: direct fix-and-verify with operator surface-review checkpoint before commit
     (11+ pages touched visually; no behaviour change).

10. **Binding ADR number:** brief allocated ADR-0042; pre-polish audit consumed it; binding ADR
    for polish stage = ADR-0043 (this document).

## Consequences

- Positive: all 15 in-scope issues close in a bounded 1–2 day window; +27 min test floor;
  stage is fully planned before first line of code.
- Negative: Cluster A is a hard dependency for Clusters B and (partially) F; cannot parallelise
  these until A lands.
- Follow-ups: ISSUE-0038 (axe live) resolves automatically at preview deploy gate.
  ISSUE-0060 (partition RLS) requires Q-1.1-AUDIT-1 operator answer before closing.
  ISSUE-0067 README workaround note required in a polish-stage commit or follow-up chore.
