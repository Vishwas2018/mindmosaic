# v1.1 Phase Plan — Exam Content Platform + Content Operation

Status: planning artefact. Authored 2026-05-14. References DEV-20260514-1
(v1.1 phase reprioritisation ahead of DEV_PLAN §5.1 P1.1–P1.7 backlog).

This plan supersedes the informal 5-stage outline. Stages S1–S5 build the
_platform_ (machinery). Stages S6–S7 run the _content operation_ (data).
The two are deliberately separated: the platform handles any volume of
content; the content operation fills it. Building S6 tooling before S1–S5
machinery exists would be premature scope — S6+ stays parked until S5 closes.

## Part 2 — Content Operation Stages (S6–S7)

PARKED until S5 closes. The platform must exist before content can live in
it. These stages turn the empty machinery into a populated product.

### Critical constraint — copyright and trademark

NAPLAN and ICAS are real, copyrighted assessment products with protected
names. The content operation produces **original items only** — written
fresh to match the _format, year-level curriculum strand, difficulty band,
and question type_ of those assessments, without reproducing any actual
past-paper item. No scraping of past papers, no licensed-bank ingestion,
no copying. This constraint shapes every part of S6 and S7. It is not
optional and not negotiable; treat it with the same weight as the child-
safety and security non-negotiables in CLAUDE.md.

Practical implications:

- Item-authoring spec templates describe format and curriculum alignment,
  never specific real items.
- "NAPLAN-style" / "ICAS-style" is the framing — alignment, not copy.
- Any generated item that too-closely paraphrases a known past-paper item
  is rejected at review. The lifecycle FSM `draft → review` gate exists
  partly for this.
- Legal review of the authoring spec templates before bulk authoring runs
  is a hard prerequisite, not a nicety.

### S6 — Bulk Content Import Pipeline + Authoring Spec Templates (~3 days)

**Status: complete (28e85e2 impl · this chore close)**

The tooling stage. Builds the leverage point for everything in S7.

Deliverables:

- **Batch import endpoint / script** — `POST /content/items/batch` or a
  `scripts/import-items.ts` consuming a strict JSON manifest matching the
  `item` / `item_version` / `stimulus` schema. Idempotent (re-runnable;
  dedupe on a manifest-supplied external key). Validates every item
  against Zod schemas + curriculum-strand enums before any DB write.
  Partial-failure reporting (item N rejected, reason) — never a silent
  half-import.
- **Authoring spec templates** — `docs/content/specs/` — one per
  assessment × year × subject strand. Each template defines: target
  format, curriculum strand (mapped to skill-graph nodes), difficulty
  band definitions, question-type distribution, worked-solution
  requirements, distractor-rationale requirements, and explicit
  copyright guardrails ("original items only; alignment not reproduction").
  These templates are the contract Claude (or human SMEs) author against.
- **Import validation gate** — schema validation + curriculum-strand
  coverage check + a duplicate-similarity check (flag items that are
  near-duplicates of existing bank items or of each other).
- **Manifest format spec** — `docs/content/manifest-format.md` — the
  exact JSON shape, versioned.
- ADR-00XX — content-import pipeline patterns (idempotency model,
  validation gate, partial-failure semantics).
- Contract tests for the import pipeline (happy path, partial failure,
  duplicate rejection, schema violation, idempotent re-run).

S6 ships **no actual content**. It ships the pipeline and the templates.

### S7 — Content Authoring & Bank Population (ONGOING — not a fixed stage)

The operational stage. Runs the S6 pipeline to populate the bank.
This is not a "complete in 2 days" stage — it is ongoing operational
work that continues past v1.1 launch. Sequenced as sub-batches:

Recommended sub-batch order (smallest complete slice first):

- **S7.1** — One pilot batch: NAPLAN-style Year 5 Numeracy, ~50 original
  items, full lifecycle (draft → review → active). Proves the end-to-end
  loop: author against S6 template → import → review → activate →
  composable by S2 → deliverable by S3/S5. Do not scale until S7.1 works.
- **S7.2** — NAPLAN-style remaining Year 5 subjects, then Years 3, 7, 9.
- **S7.3** — ICAS-style, by subject and year.
- **S7.4+** — ongoing: monitor item performance via the `monitored`
  lifecycle state (difficulty drift, low discrimination), retire weak
  items, author replacements.

Authoring approaches (choose per batch; hybrid recommended):

- **AI-assisted (Claude):** Claude drafts original items to an S6 spec
  template, output as import-manifest JSON. Fast at volume. Every item
  still passes human review before `active`.
- **Human SME:** subject-matter experts author to the same templates.
  Slower, higher baseline quality.
- **Hybrid (recommended):** Claude drafts → human SME reviews and approves
  → lifecycle FSM tracks state. Claude does volume; humans gate quality.
  The `review` and `monitored` FSM states are built for exactly this.

Volume expectation: "thousands of items" is realistic for the hybrid
approach over time, but S7.1 (50 items, one strand) must succeed first.
Quality gate is non-negotiable: an item bank with weak items is worse
than a smaller bank of good ones — weak items corrupt difficulty
calibration and student experience.

S7 has no single "done." A question bank is seeded, launched, monitored,
and continuously tended. Track progress as a coverage matrix
(assessment × year × subject × item-count × lifecycle-state) in
`docs/content/coverage.md`, updated per sub-batch.

## Why S6–S7 stay parked

Building import tooling before the CRUD endpoints exist, or authoring
content before there is a bank to hold it, is exactly the premature-scope
and speculative-engineering failure mode the discipline framework guards
against. S6 depends on S1's schema being final and S2/S3's consumption
contracts being known (so the manifest format matches what composers and
simulators actually read). S7 depends on S6's pipeline existing. The
dependency chain is real, not bureaucratic — respect it.

## Open decisions deferred to S6 morning ritual

- Batch import as an authenticated endpoint vs a service-role script.
- Manifest external-key scheme for idempotency.
- Whether curriculum-strand → skill-graph-node mapping needs a new
  lookup table or fits existing `skill_node` tagging.
- Duplicate-similarity check: exact-match only (cheap) vs fuzzy
  (needs an embedding or shingle approach — possible scope expansion).
- Legal review process and owner for authoring spec templates.

These are flagged, not resolved. They get T3 round-trips at S6 start —
which is after S5, not now.
