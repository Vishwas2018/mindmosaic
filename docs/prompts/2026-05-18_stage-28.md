## Q-28 Resolutions (all resolved pre-implementation)

**Q-28.1 — ISSUE-0006 in scope**: YES. Replace direct `skill_edge` query in
`intelligence-svc/handlers.ts` L3a with `getSkillGraph()` call from
`_shared/skill-graph-cache.ts`. Required: L3b also reads via `getSkillGraph()` per
ADR-0028.

**Q-28.2 — HTTP architecture**: jobs-worker = generic runtime; each `job_type` maps
to owning service via HTTP. Recorded in ADR-0031. `pipeline.causal.evaluate_full` →
`POST /intelligence/pipeline/causal-full` (service-role + `x-mm-trace-id`). Retry +
backoff state in jobs-worker only, not in intelligence-svc handler.

**Q-28.3 — job_queue schema**: Read the Stage 9 migration (`supabase/migrations/
0009_job_queue_cron.sql` or equivalent) first. If `attempt_count`, `max_attempts`,
`dead_lettered_at`, `failure_reason`, `next_attempt_at` columns are absent, write an
ALTER TABLE migration — never assume they exist.

**Q-28.4 — job_queue column additions**: Default column set approved: `status` text
enum (`pending` | `processing` | `completed` | `failed` | `dead_lettered`),
`dead_lettered_at timestamptz`, `failure_reason text`. Add in a new migration if
absent from the Stage 9 schema.

**Q-28.5 — Test pattern**: Vitest-with-mock (node-compatible) for the named
500-jobs test; PLUS one opt-in real-Postgres integration test guarded with
`test.skip(process.env.DOCKER_AVAILABLE !== '1', ...)`. Named tests required:
- `'500 jobs enqueued, all processed, zero duplicates'`
- `'failure injection drives retries then dead-letter'`

**Q-28.6 — L3b traversal depth**: Read spec §5.1.3 + §5.1.4 before writing code for
`traverse_upstream` / `traverse_downstream`. Cycle detection is non-negotiable (graph
may have cycles from content authoring). Default safety cap: 50 nodes per direction.
On cap hit: log structured warn (`{ event: 'skill_graph_cycle_cap_hit', skill_id,
direction, visited_count }`) and return the partial set — do NOT throw.
