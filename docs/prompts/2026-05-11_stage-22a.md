## Context

Stage 22 (Day 27) was scoped as Session Selection + Practice screens.
At implementation start, the §2A walkthrough surfaced two pre-existing
architectural gaps the screens depend on:

1. **`MmClient` baseUrl strategy is undefined for multi-service Edge
   Function deployment.** Every Edge Function lives at
   `${SUPABASE_URL}/functions/v1/<svc-name>/<path>`. SDK hooks pass
   service-bare paths like `/sessions/recent`. No prior ADR pins how
   bare paths resolve to per-service URLs.
2. **Several SDK hook paths do not match their Edge Function
   dispatcher's actual route.** Confirmed (spot-grep, not full audit):
   `useCreateSession` calls `POST /sessions` but
   `assessment-svc/index.ts:217` serves `POST /sessions/create`;
   `useSessionSummary` calls `GET /sessions/{id}/summary` but
   `assessment-svc/index.ts:352` serves `GET /sessions/{id}`. More
   may exist.

Both gaps were surfaced by exactly the conditions the green-light
prompt warned of ("first stage to wire SDK hooks into apps/web routes;
watch for SDK hook gaps surfaced by real route consumption"). Building
screens against broken SDK plumbing would compound the gap; fixing
both is a discrete piece of infrastructure that is the honest content
of today's day. **Stage 22 split into 22a (today, infrastructure) +
22b (tomorrow, screens)**, costing −1 Phase 1 buffer day per
DEV-20260511-1.

ADR-0029 records the architectural decision: **single `MmClient`
rooted at `${SUPABASE_URL}/functions/v1`; each hook prepends its
service prefix in the path**. No mapping table, no proxy layer, no
per-service providers. Q-22.2 records the mechanical-correction
follow-up: dispatcher paths win; SDK paths are patched to match the
route each dispatcher actually serves.

## Deliverables

### 1. Full audit table (report-only, before edits)

Grep `client.get(`/`client.post(`/`client.put(`/`client.delete(`
across `packages/sdk/src/hooks/` and cross-reference each path
against the `path === '...'` and `path.match(/^.../)` lines in
`supabase/functions/{assessment-svc,content-svc,intelligence-svc}/index.ts`.

Output a table: SDK file:line | SDK path | Dispatcher path | Match?

This pauses for review before any edit.

### 2. SDK path corrections (Q-22.2 + ADR-0029)

For every hook in `packages/sdk/src/hooks/*.ts`:

- Prepend the canonical service prefix (`/assessment-svc`,
  `/content-svc`, `/intelligence-svc`,
  `/orchestration-svc` for not-yet-shipped hooks).
- Where the bare path doesn't match the dispatcher's route, correct
  the path component (e.g. `/sessions` → `/sessions/create`;
  `/sessions/{id}/summary` → `/sessions/{id}`).
- Add a one-line file-header comment naming the service the file
  targets, e.g. `// hooks/session.ts → assessment-svc`.

### 3. SDK test path corrections

`packages/sdk/src/__tests__/hooks.test.ts`:
- The existing `useMe` plumbing test asserts a fetch URL match;
  update the regex to expect the new full path (with service prefix).
- The new `useListRecentSessions` test (added earlier today) already
  asserts `/sessions/recent$`; update to expect
  `/assessment-svc/sessions/recent$`.
- Do not add new tests beyond these path-correction updates.

### 4. `MmClient` default / construction site

`packages/sdk/src/client.ts` — no API change required (baseUrl is
already a string config). Document in a header comment that callers
should set `baseUrl = ${SUPABASE_URL}/functions/v1` per ADR-0029
and that hooks own their service prefix.

### 5. `MmClientProvider` wired into apps/web

`apps/web/src/providers/Providers.tsx`:
- Instantiate `MmClient` once with
  `baseUrl = ${NEXT_PUBLIC_SUPABASE_URL}/functions/v1` and
  `getToken = () => session?.access_token ?? null` (pulled from
  the existing AuthProvider's session state).
- Mount `<MmClientProvider client={client}>` between
  `EntitlementsProvider` and `TooltipProvider` (or wherever the
  tree is logically deepest before consumer hooks run — order is
  not behavioural, only nesting depth matters).
- The `MmClient` instance must be stable across renders (use
  `useMemo` keyed on the access token) so React Query keys stay
  consistent.

### 6. No `apps/web/.env.local.example` change

ISSUE-0005 stays deferred. The `NEXT_PUBLIC_SUPABASE_URL` env var
is already populated. No new env var required.
