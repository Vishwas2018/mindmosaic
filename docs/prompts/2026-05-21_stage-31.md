## Constraints

### Spec-pinned values (T1 — verbatim from spec §16):

- `generate_weekly_plan(student, available_minutes_per_week):` — no default; derived at runtime
  as `session_length_sweet_spot * 5` (Q-31.1 resolution)
- Priority order: `repair > root cause > declining > low retention > practice > stretch`
- Declining velocity threshold: `velocity < -0.01` (spec §16.2, line 2331)
  **NOT -0.02** — that was the L7 trigger (§14.2, different spec section)
- Low retention condition: `retention_estimate < 0.5 and mastery > 0.6` (§16.2, line 2335)
  `retention_estimate` computed inline as `mastery_level * exp(-(days_since_last_attempt /
  retentionHalfLifeDays(year_level)))` using `@mm/engines/src/constants/retention.ts`
  (Stage 29, Q-29.3). Values: `{y5: 60, y7: 90, y9: 120, default: 90}`. Do NOT hardcode 14d.
  NULL `last_attempted_at` → `retention_estimate = mastery_level` (no decay).
- Session guardrail: `behaviour.session_length_sweet_spot + 5 min` max per session (§16.6)
- Max concurrent repairs: `No more than 3 repair sequences active simultaneously` (§16.6)
- Enjoyment minimum: `At least 20% of plan time is "enjoyment" sessions` (§16.6)
- Override default expiry: `14 days if not specified` (§16.6.1, line 2504)
- Override audit: `every override creation and expiry writes an entry to intelligence_audit_log
  with layer='L9_override'` (§16.6.1, line 2508)
- Interleave rule: `no more than 2 consecutive sessions on same domain` (§16.2, line 2358)
- Idempotency skip: `learning_plan.updated_at > job scheduled_at` (DEV_PLAN verbatim)
- Enum: `plan_type = 'weekly' | 'exam_countdown' | 'long_term' | 'transition'` (migration 0001)
- Enum: `plan_status = 'active' | 'superseded' | 'expired'` (migration 0001)

### Idempotency test:

Handler called twice with same `{student_id, session_id, scheduled_at}`:
second call checks `updated_at > scheduled_at` after first call's INSERT → early return, zero
additional DB writes. First write must set `updated_at` to a value > `scheduled_at`.

### Override tests:

- `pin_skill` override → target skill appears in plan at specified priority, even if it would
  not have appeared from intelligence signals alone
- `dismiss_recommendation` → skill filtered from queue by recommendation key match

### Failure path test:

Handler throws during plan assembly → existing active plan has `stale_since` set (UPDATE only),
no new `learning_plan` row, no `plan_revision` row.
