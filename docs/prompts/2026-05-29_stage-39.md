## Constraints

C1. All assignments-svc backend endpoints shipped Stage 33. All analytics-svc generate-assignment shipped Stage 32. No Deno/Edge Function changes in Stage 39.

C2. pathway_id in CreateBody is a UUID FK → pathway(id). Source: usePathways().data[0].id (NAPLAN is the only active v1 pathway). PathwayDTOSchema additive fix (Q-39.5, D1) is committed BEFORE any useCreateAssignment hook code (D3). T2-tightened sequence enforced.

C3. DraftAssignmentDTO has no pathway_id. Wizard tracks selected pathway_id independently from the generate-assignment response. pathway_id is never null in v1 (first pathway from usePathways()).

C4. Idempotency-Key header sent from useCreateAssignment and usePublishAssignment mutations (client generates crypto.randomUUID()). Server accepts and logs; no dedup enforcement (DEV-20260523-1, ISSUE-0023). Q-39.1 resolution.

C5. targets in CreateBody = Array<{type:'student'|'class', id:string}>. Wizard mapping (Q-39.3):
    - "Entire Class" → [{type:'class', id:classId}]
    - ?target_student={uuid} or Custom Selection → [{type:'student', id:studentId}]
    - At-Risk → up to 3 {type:'student'} from useTeacherRefresh intervention_alerts; fallback to class if unavailable.

C6. v1-no-op fields render in DISABLED state with tooltip "Available in a future release" (PHASE-2 pattern — NOT silent-dropped, UX-honest):
    - Attempts Allowed select (Configure step) — no DB column
    - Start Date input (Schedule step) — no DB column
    - Send Reminder checkbox (Schedule step) — SCREEN_SPECS explicit v1.1 deferral

C7. Topic chips caption (below chip strip, before AutoSuggestBanner): "Suggestions only — auto-generated based on class analytics." generateAssignment called with {class_id, mode} only (no target_skill_ids filter). Draft provides target_skill_ids for subsequent createAssignment. Q-39.UI-3 resolution.

C8. Mode mapping (Q-39.6): type card keys used directly as CreateBody.mode: practice | diagnostic | exam | skill. No blueprint_based in v1 — the "Blueprint-based" type card is NOT rendered (no DB-backed path; entirely absent, not disabled).

C9. Tab → DB status mapping (list view): Active = status='published'; Upcoming = status='draft'; Completed = status='archived'.

C10. 4-item sidebar nav (Overview / Students / Assignments / Analytics). Insights NOT added (Q-39.UI-6 — no spec backing).

C11. Stale-comment guard. Do not reference Stage 36–38 variable names in new files:
    - Stage 36: child, parent, useChildRecentSessions, useMyChildren
    - Stage 37: ClassKpiStrip, InterventionAlertsSection, useAssignmentsForClass (exists; may import but don't duplicate)
    - Stage 38: TeacherNotes, StrandMasteryCard, useFlagForReview (exists; don't redeclare)
    - No `any` in production code (TypeScript strict).

C12. PHASE-2 comment guard: any feature beyond v1 scope must carry `// PHASE-2:` comment and "Available in a future release" UI copy (grep-detectable).

C13. Pre-push verification round MANDATORY before push. Surface V1–V8 to architect before any git push. Stage 38 retro reinstatement.

C14. Auto-generate flow integration: fires silently after teacher completes step 1 (Target) and presses Continue. useGenerateAssignment mutation called with {class_id, mode}. On success: pre-populates step 2 Configure values from DraftAssignmentDTO (item_count → questions, difficulty_range → difficulty). On failure: step 2 shows defaults (15 questions, Mixed difficulty); no error shown to teacher (non-blocking degradation). Draft target_skill_ids stored in wizard state for the eventual createAssignment call.
